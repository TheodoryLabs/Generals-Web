## The one open hazard: the Chrome stable V8 GC bug

Chrome 149 (< .201) and 150 (<= 150.0.7871.47) stable can kill the renderer
of any large wasm app, probabilistically, most often at match start. Fixed
in V8 14.9.207.37 / 15.0.245.14. Everything we control is mitigation:
less allocation churn (done), fewer/smaller sync-XHR allocations (done),
user guidance + `scripts/play-web.sh` escape hatch (done).

**Watch:** when a Chrome 150 respin with V8 >= 15.0.245.14 reaches stable
(check `chromiumdash.appspot.com/fetch_releases?channel=Stable&platform=Mac`),
retest without any flags, then soften the README Known Issues section.

## Hard-won lessons (do not relearn these the expensive way)

- One clean run proves nothing here — the crash is probabilistic. Only
  repeated counted trials count.
- `--js-flags=--no-compact` is a partial mitigation only, and is silently
  ignored if Chrome is already running when you launch.
- Chrome applies downloaded updates only on full relaunch; verify the
  *running* version from the page's UA, never from the About box.
- Diagnosis instrumentation must be allocation-light or it changes the
  phenomenon (console-wrapping loggers measurably raised crash odds).
- The old builds' log spam accidentally paced the fetch storm; "removing
  overhead" can remove an accidental safety margin.
